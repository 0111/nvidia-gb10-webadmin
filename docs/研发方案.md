## 2. 核心模块说明

### 2.1 core/config.py

- `AppConfig`（dataclass）：`web_port`(8000), `web_host`("0.0.0.0"),
  `secret_key`(随机生成), `admin_username`("admin"),
  `admin_password`(随机生成), `model_root_dir`
  ("/home/spark/LocalModels/LLModels"),
  `data_dir`("/home/spark/nvidia-gb10-manager/data"), `searxng_port`(8080),
  `searxng_url`, `vllm_image`("nvcr.io/nvidia/vllm:26.06-py3"),
  `cuda_compat_dir`("/usr/local/cuda-13.3/compat")。
- `load_config(path) -> AppConfig`：文件不存在时返回带随机密钥的默认配置（不写盘）。
- `save_config(config, path)`：YAML 写盘，自动创建父目录。
- 默认路径：`config/settings.yaml`（样例见 `settings.example.yaml`）。

### 2.2 core/env_doctor.py

统一返回结构：

- `CheckResult(name, status, message, suggested_command, fixable, details)`，
  `status` ∈ {ok, warning, error}。
- `FixResult(name, executed, command, stdout, stderr, returncode, message)`。
- `EnvReport(checks: list[CheckResult])`，`overall_status` 属性按
  error > warning > ok 汇总；`to_dict()` 供 Web JSON 序列化。

检测函数（均无副作用）：

| 函数 | 检测依据 | 默认阈值 |
| --- | --- | --- |
| `check_cuda_compat(compat_dir=/usr/local/cuda-13.3/compat)` | 目录是否存在 + libcuda.so/libcuda.so.1 是否存在 | - |
| `check_ethernet_speed(interface=None)` | 读 `/sys/class/net/<iface>/speed`，失败时回退 `ethtool` | < 1000 Mb/s 视为 warning |
| `check_drop_caches()` | 解析 `/proc/meminfo` 的 Cached/MemTotal 比例 | 比例 ≥ 0.40 (`CACHED_WARNING_RATIO`) 视为 warning |
| `check_swap()` | 解析 `/proc/swaps` 是否有有效条目 | 有条目即 warning |
| `run_all_checks()` | 按 cuda_compat -> ethernet_speed -> drop_caches -> swap 顺序汇总 | - |

修复函数（均默认 `confirmed=False`，仅返回将执行的命令字符串；
`confirmed=True` 时才真正 subprocess 执行）：

- `fix_cuda_compat(confirmed=False)` → `sudo apt install -y cuda-compat-13-3`
- `fix_ethernet_speed(interface, confirmed=False)` → `sudo ethtool -s <if> speed 1000 duplex full autoneg on`
- `fix_drop_caches(confirmed=False)` → `sync && echo 3 | sudo tee /proc/sys/vm/drop_caches`
- `fix_swap(confirmed=False)` → `sudo swapoff -a`

### 2.3 core/model_scanner.py

- `ModelInfo`（dataclass）：`name, path, format(safetensors/gguf/unknown),
  size_bytes, is_embedding, quantization, max_position_embeddings,
  architectures, tool_call_capable, engine_hint(vllm/ollama/unknown),
  warnings`。
- `scan_models(root_dir) -> list[ModelInfo]`：对 `root_dir` 下所有目录做
  `rglob`，只要目录直接包含 `config.json` 或任意 `*.gguf` 文件即视为一个
  模型；单目录解析异常只记录 warning，不中断整体扫描。
- 关键判定逻辑：
  - `is_embedding`：路径/目录名含 embedding/embed/reranker/rerank 关键字，
    或 config.json 的 `model_type`/`architectures` 命中同样关键字。
  - `quantization`：优先匹配目录名中的 NVFP4/FP8/MXFP4/INT4/INT8/AWQ/GPTQ；
    否则读取 `hf_quant_config.json` 的 `quant_algo`；否则读取
    `config.json.quantization_config.quant_method`；GGUF 文件则从文件名
    提取 Q8_0/Q6_K 等标签。
  - `tool_call_capable`：存在 `*tool_parser*.py` 文件，或
    `chat_template.jinja` 内容含 "tool_call"/"tools"。
  - `engine_hint`：safetensors → vllm，gguf → ollama，否则 unknown。
- `list_general_models(models)` / `list_embedding_models(models)`：按
  `is_embedding` 过滤。

### 2.4 core/param_advisor.py

- `recommend_vllm_params(model_info) -> dict[str, {default, editable,
  options, source}]`，覆盖 Project_Task.md 2.3.3 列出的全部参数：
  served_model_name, gpu_memory_utilization, max_model_len, max_num_seqs,
  max_num_batched_tokens, enable_auto_tool_choice, tool_call_parser,
  chat_template, enable_chunked_prefill, enable_prefix_caching,
  trust_remote_code, kv_cache_memory_bytes, kv_cache_dtype, tokenizer,
  dtype, quantization, generation_config, reasoning_parser,
  tokenizer_mode, task, runner。
  - `max_model_len` / `max_num_batched_tokens` 下拉选项
    `[4k,8k,16k,32k,64k,96k,128k,160k,256k]`，并被模型自身
    `max_position_embeddings` 截断（`_context_len_options`）。
  - `max_num_seqs` 下拉 `[1,2,3,4]`，默认 1。
  - `enable_auto_tool_choice` 默认 `False`；`editable` 仅当
    `model_info.tool_call_capable` 为真时才为 `True`。
  - embedding 模型强制 `task="embed"`、`editable=False`、
    `runner="pooling"`。
- `recommend_ollama_params(model_info) -> dict`：最小可用占位版本
  （num_ctx/num_predict/num_gpu/num_thread/num_batch/temperature/
  quantization），注释里明确标注需要在部署前对照 ollama 实际版本复核。

### 2.5 core/docker_helper.py

- `DockerComposeManager(model_root_host, cuda_compat_dir)`
  - `render_compose(model_info, vllm_params, image, host_port,
    container_port) -> str`：用字符串模板渲染 compose 内容，包含
    `LD_LIBRARY_PATH` 环境变量与 cuda-compat 只读卷挂载，命令行参数从
    `vllm_params` 的 `default` 值拼装。
  - `start/stop/restart/status/down/logs(compose_path, ...) ->
    CommandResult`：均通过 `subprocess.run(["docker", "compose", "-f",
    ...])` 实现，异常（FileNotFoundError/Timeout/其他）全部捕获，返回
    `CommandResult(ok, command, stdout, stderr, returncode, error)`，不会
    抛出未捕获异常。

### 2.6 cli/main.py

- 基于 `click.group`，全局选项 `--config`（默认指向
  `config/settings.yaml`）。
- 子命令：`start / stop / restart / clean / model_check`。
- `run_env_checklist(auto_fix_prompt=True)`：打印
  `env_doctor.run_all_checks()` 结果；对每个 `fixable` 且非 ok 的检查项，
  用 `click.confirm()` 询问是否立即修复，确认后才调用对应 `fix_*`
  （`confirmed=True`）。
- `model_check`：调用 `scan_models` + `list_general_models` /
  `list_embedding_models`，打印清单，并写入
  `<data_dir>/model_scan_result.json`。

## 4. 测试用例设计建议（供测试经理参考）

### core/config.py

- 测试：`load_config` 在文件不存在时返回默认值且不抛异常。依据：
  `load_config` 实现里 `if not path.exists(): return AppConfig()`。期望：
  返回的 `AppConfig` 各字段为类定义的默认值，`secret_key`/`admin_password`
  非空。
- 测试：`save_config` 后再 `load_config` 应得到字段相同的对象（secret_key
  等随机值应保持不变，验证未被二次随机化）。依据：YAML 往返序列化。期望：
  两次对象 `to_dict()` 相等。
- 测试：配置文件中包含未知字段时 `load_config` 不报错（向后兼容）。依据：
  `known_fields` 过滤逻辑。期望：未知字段被忽略，已知字段正确加载。

### core/env_doctor.py

- 测试：`check_cuda_compat` 指向一个不存在的临时目录时返回 `status=error`
  且 `suggested_command` 含 `apt install cuda-compat-13-3`。依据：函数实现
  的第一分支。期望：`fixable=True`。
- 测试：`check_cuda_compat` 指向一个包含 `libcuda.so`/`libcuda.so.1` 的临时
  目录时返回 `status=ok`。依据：required 文件名单。期望：`fixable` 字段
  未设置（默认 False）。
- 测试：`check_drop_caches` 在 mock `/proc/meminfo` 内容下，
  Cached/MemTotal ≥ 0.40 时返回 warning，否则 ok。依据：
  `CACHED_WARNING_RATIO` 常量。期望：`details.cached_ratio` 数值正确。
- 测试：`check_swap` 在 mock `/proc/swaps` 只有表头行时返回 ok；有数据行时
  返回 warning 且 `suggested_command == "sudo swapoff -a"`。
- 测试：所有 `fix_*` 函数在 `confirmed=False`（默认）时，不应有任何
  `subprocess.run` 调用发生（可用 mock 断言未被调用），且
  `executed=False`。依据：安全约束——危险操作需显式确认。这是本模块最
  关键的验收点。
- 测试：`run_all_checks()` 返回的 `EnvReport.overall_status` 在任一子检查
  为 error 时整体为 error，否则任一为 warning 时整体为 warning，否则 ok。

### core/model_scanner.py

- 测试：对 Project_Task.md 1.3 节给出的示例目录树（构造临时目录模拟
  Deepseek/Gemma/Qwen/Embedding 等结构）运行 `scan_models`，验证返回模型
  数量与预期一致，且不因为某个模型缺少 config.json（如纯 GGUF 目录）而
  抛出异常。依据：需求里"优雅降级"的硬性要求。
- 测试：`Qwen/Embedding/Qwen3-VL-Embedding-2B` 这类路径应被判定为
  `is_embedding=True`；`Qwen/Qwen3.6-27B` 应为 `False`。依据：关键字匹配
  逻辑覆盖路径含 "Embedding"。
- 测试：`DeepSeek-R1-Distill-Qwen-32B-NVFP4` 目录应解析出
  `quantization="NVFP4"`；`GLM-4.7-Flash-FP8` 应解析出 `"FP8"`；纯 GGUF
  目录（如 `Gemma-4-31B-JANG_4M-CRACK-GGUF`）应解析出形如 `"GGUF Q8_0"`
  的值。依据：目录名标签匹配 + 文件名提取兜底两条路径都要覆盖。
- 测试：`Qwen3-Coder-30B-A3B-Instruct-FP8`（含
  `qwen3coder_tool_parser.py`）应得到 `tool_call_capable=True`；普通模型
  目录（无 tool_parser 文件、chat_template 不含 tool 关键字）应为
  `False`。
- 测试：config.json 内容为非法 JSON 时，`ModelInfo.warnings` 应包含
  "config.json 解析失败"，且其余字段仍尽量填充（如 format/size_bytes 不
  受影响）。依据：try/except 包裹 + 不中断整体扫描。
- 测试：`list_general_models`/`list_embedding_models` 对同一份
  `scan_models` 结果应互补且无交集（并集等于全集）。

### core/param_advisor.py

- 测试：`max_model_len` 推荐值不得超过 `model_info.max_position_embeddings`
  （当后者已知时）；当后者大于所有下拉档位时，应取下拉清单最大档位
  （262144）。依据：`_context_len_options` 的截断逻辑与"挑选最接近模型上限
  的档位"需求。
- 测试：`model_info.tool_call_capable=False` 时，
  `enable_auto_tool_choice` 与 `tool_call_parser` 的 `editable` 必须为
  `False`，无论调用方是否尝试覆盖；为 `True` 时才允许编辑。依据：
  Project_Task.md 2.3.3 节"如果不支持则不允许用户选择"的硬性约束——这是
  最重要的安全/正确性测试点。
- 测试：`model_info.is_embedding=True` 时，返回字典中 `task` 必须固定为
  `"embed"` 且 `editable=False`，`runner` 默认 `"pooling"`；
  `is_embedding=False` 时 `runner` 默认 `"generate"`。
- 测试：`max_num_seqs` 选项固定为 `[1,2,3,4]`，默认值 `1`，不随模型变化。
- 测试：`recommend_ollama_params` 对 gguf 格式模型应返回非空字典且包含
  `num_ctx`，其默认值同样受模型上限截断（若可解析到）。

### core/docker_helper.py

- 测试：`render_compose` 输出字符串中必须包含
  `LD_LIBRARY_PATH=/usr/local/cuda/compat:...`、
  `"{cuda_compat_dir}:/usr/local/cuda/compat:ro"`、以及
  `served-model-name` 参数行。依据：Project_Task.md 1.2 节给出的环境变量
  与卷挂载示例必须原样体现。
- 测试：`render_compose` 在 `vllm_params` 某些条目 `default=None`
  （如 `tokenizer`）时不应在生成的 command 列表中出现对应 flag（避免拼出
  `--tokenizer None`）。
- 测试：`DockerComposeManager.start/stop/restart/status` 在 `docker`
  命令不存在（mock `FileNotFoundError`）时应返回 `CommandResult(ok=False,
  error="docker 命令未找到")`，不抛出异常导致调用方崩溃。依据：需求里
  "异常要捕获不要让程序崩溃"。
- 测试：以上四个方法在 `subprocess.run` 返回非 0 `returncode` 时，
  `ok=False` 且 `stderr` 被正确传递。

### cli/main.py

- 测试（手动/集成，需在 DGX Spark 上验证）：`model_check` 命令运行后，
  `data/model_scan_result.json` 文件存在且 JSON 可解析，其
  `general_models`/`embedding_models` 数量之和与终端打印的"共发现 N
  个模型"一致。
- 测试：当某个 env check 状态为 `warning`/`error` 且 `fixable=True` 时，
  CLI 在用户输入 `n`（拒绝修复）后，不应有任何系统状态被改变（即不应执行
  对应的 `fix_*` 真实命令）。依据：交互确认必须真正阻断危险操作。
- 测试：`start`/`restart` 命令打印的"运行信息"中端口、API Base URL、
  密钥（截断显示）三项均不为空。

## 1. 目录结构与模块作用

```
web/
├── __init__.py
├── main.py                  # FastAPI app 入口：挂载路由、CORS、startup/shutdown 事件、/api/health
├── auth.py                   # 用户名+密码校验、JWT 签发(create_access_token)与校验(verify_token)、
│                              # get_current_user 依赖注入、verify_token_for_ws（WS 专用、失败返回 None 而非抛异常）
├── ws_hub.py                 # ConnectionManager：connect/disconnect/update_topics/broadcast(topic, data)
│                              # 单连接发送失败会被捕获并剔除该连接，不影响其他连接的广播
├── ws_router.py               # WebSocket 端点 GET /ws，查询参数 token + topics
├── background_tasks.py        # MetricsCollector：asyncio 后台循环，每10秒采集 CPU/内存(psutil)+GPU(pynvml)
│                              # 并广播到 "metrics" topic；任一库缺失/读取失败时优雅降级为 None，不抛异常
├── state.py                   # 进程内状态：ModelLoadRegistry 记录通过本系统加载的模型→compose文件→容器名映射
│                              # （重启后丢失，符合"不持久化、需人工启动"的项目原则）
├── schemas.py                 # 所有请求/响应 Pydantic 模型，字段 snake_case，并提供 from_core() 适配器
│                              # 将 core 的 dataclass（CheckResult/ModelInfo/FixResult/CommandResult等）转为响应体
└── routers/
    ├── auth_router.py         # POST /api/auth/login
    ├── overview_router.py     # GET /api/overview
    ├── models_router.py       # GET /api/models, GET /{name}/params, POST /{name}/load|unload
    ├── components_router.py   # GET /api/components, POST /{name}/{start|stop|restart}
    ├── env_router.py          # GET /api/env/checklist, POST /api/env/fix/{check_name}
    ├── logs_router.py         # GET /api/logs/{component}
    ├── debug_router.py        # POST /api/debug/chat
    └── settings_router.py     # GET/PUT /api/settings
```

设计要点：

- **鉴权边界**：除 `auth_router`（登录）和 `/api/health`（健康检查）外，
  所有 REST 路由都在 `APIRouter(..., dependencies=[Depends(get_current_user)])`
  级别统一保护，无需在每个端点函数上重复声明。
- **危险操作确认机制延续阶段一**：`env_router.fix_check` 沿用
  `core.env_doctor` 的 `confirmed` 语义——请求体 `confirmed != true` 时直接
  返回 400，并提示"需要确认"，**不会**调用任何 `fix_*` 函数，因此不存在
  误触发 `apt install`/`swapoff`/`drop_caches`/`ethtool -s` 的路径。
- **WebSocket 鉴权**：浏览器原生 WebSocket API 无法自定义请求头，因此采用
  `?token=<jwt>` 查询参数传递 JWT；校验失败时用 close code `4401` 主动关闭
  连接（而不是静默断开或抛 500），便于前端区分"未授权"与"网络异常"。
- **模型加载状态的进程内登记**：`core/docker_helper.py` 本身不维护"哪个
  模型已加载"的状态，本阶段在 `web/state.py` 新增轻量 `ModelLoadRegistry`
  （仅内存，不落盘）记录 `模型名 -> compose 文件路径/容器名/端口`，供
  `overview_router`/`components_router`/`logs_router`/`models_router.unload`
  共用，避免每个路由各自维护一份状态。这是新增内容，不影响
  `cli/main.py` 对 `DockerComposeManager` 现有方法的调用方式。
- **core/docker_helper.py 未做破坏性改动**：Web 后端直接调用其现有的
  `render_compose/start/stop/restart/status/down/logs` 方法，未新增/修改
  其签名；`models_router.load_model` 把前端提交的 `{参数名: 值}` 扁平字典
  适配成 `render_compose` 期望的 `{参数名: {"default": 值}}` 形状，适配逻辑
  留在 Web 层，没有侵入 core。

## 3. WebSocket 设计

- 端点：`GET /ws?token=<jwt>&topics=metrics,load_progress`（topics 可逗号分隔多个，默认 `metrics,load_progress`）。
- 鉴权：连接建立（accept）后立即校验 token；无效时以 close code `4401` 关闭。
- 消息包络（服务端→客户端，所有广播统一格式）：
  ```json
  {"topic": "metrics", "data": { ... }, "ts": 1719400000.123}
  ```
- Topic 清单：
  - `metrics`：每 10 秒广播一次，`data` 形如
    `{cpu_percent, mem_used_gb, mem_free_gb, cache_gb, gpu_percent, gpu_temp_c, power_watts}`；
    pynvml/psutil 不可用或读取失败时对应字段为 `null`，不会中断循环或导致其它字段缺失。
  - `load_progress`：模型加载/卸载过程中由 `models_router` 主动广播，
    `data` 形如 `{model_name, stage: "starting"|"running"|"error"|"unloaded", message}`。
  - `logs`：协议位已预留（客户端可订阅），但本阶段日志仍是 REST 拉取
    （`GET /api/logs/{component}`）为主，服务端主动推送日志流是后续阶段待办。
- 客户端可在连接后发送 `{"action": "subscribe", "topics": [...]}` 动态调整订阅集合，无需重连。

## 4. 给测试经理的测试点建议

- **鉴权失败路径**：
  - 不带 `Authorization` 头访问任意受保护 REST 端点（如 `GET /api/overview`）应返回 401，且 body 不应泄露任何业务数据。
  - 使用错误用户名或错误密码调用 `POST /api/auth/login`，两种情况下错误消息必须完全一致（验证"不泄露用户名是否存在"）。
  - 使用已过期或被篡改签名的 JWT 访问受保护端点，应返回 401 而不是 500。
  - WebSocket 连接不带 `token` 或带无效 `token`，应在握手后立刻以 close code `4401` 关闭，客户端可观察到该 code。
- **危险操作确认机制**：
  - `POST /api/env/fix/cuda_compat`（以及 drop_caches/swap/ethernet_speed）在请求体缺省或 `confirmed:false` 时必须返回 400，且服务端不应有任何 `subprocess` 副作用（可在测试环境 mock `core.env_doctor.fix_*` 断言未被调用，或观察真实系统状态未变化）。
  - 同一接口 `confirmed:true` 时应真正执行并返回 `executed:true`；`ethernet_speed` 在未提供 `interface` 时应能自动从最新检查结果解析接口名，解析失败应返回 400 而非 500。
- **模型加载/卸载状态机**：
  - `POST /api/models/{name}/load` 成功后，`GET /api/models` 中对应模型 `load_status` 应变为 `running`；调用期间应能在 WS 上观察到 `load_progress` 的 `starting -> running`（或 `error`）序列。
  - 对未加载的模型调用 `unload` 应返回 404，不应误删其它模型的 compose 文件。
  - `engine="ollama"` 调用 `load` 应直接返回 400（本阶段尚未实现真实 ollama 部署模板），不应静默失败或误用 vllm 模板。
- **WebSocket 连接断开重连**：
  - 模拟客户端异常断开（不发送 close frame，直接杀连接）后，服务端下一次 `broadcast` 不应抛异常、不应影响向其它仍在线连接的推送（验证 `ws_hub.ConnectionManager.broadcast` 的"单连接失败不影响整体"设计）。
  - 客户端重连后默认应重新收到 `metrics` topic 推送（验证 topics 默认值 `metrics,load_progress` 生效）。
  - 客户端发送 `{"action":"subscribe","topics":["logs"]}` 后，应仅收到 `logs` topic 消息（如果有），不再收到 `metrics`，验证 `update_topics` 生效。
- **降级行为**：
  - 在没有 NVIDIA GPU / 未安装 pynvml 的环境下启动后端，`metrics` topic 推送的 `gpu_percent/gpu_temp_c/power_watts` 应为 `null` 而不是导致后台任务崩溃或整个进程退出。
  - `GET /api/overview` 在模型扫描目录为空或不存在时应返回空列表而非 500（依赖 `core.model_scanner.scan_models` 已有的优雅降级，需在 Web 层回归验证一次）。
- **回归（确保未破坏阶段一）**：`cli/main.py` 的 `start/stop/restart/clean/model_check` 五个子命令应在引入 `web/` 包后行为不变（`core/docker_helper.py` 接口签名未变更）。

## 4. WebSocket 使用方式

- `stores/ws.ts` 在 `App.vue` 监听 `authStore.token` 变化时调用一次
  `connect(token)`，地址为 `ws(s)://<host>/ws?token=<jwt>&topics=metrics,load_progress`，
  全局只有一条连接，所有页面通过 `useWebSocketTopic(topic, callback)`
  共享订阅，避免重复建连/重复请求。
- 断线重连：`onclose` 触发后用 `setTimeout` 固定延时 3 秒重新调用
  `connect()`（条件是 `localStorage` 中仍有 token），未实现指数退避，符合
  "内部工具、简单重试即可"的约束。
- 动态订阅：`subscribe(topic, cb)` 若发现新 topic 不在当前 `topics` 列表
  中，会通过已建立的连接发送 `{"action":"subscribe","topics":[...]}`
  消息追加订阅，无需重连（对应后端 `ws_router.py` 的 `subscribe` 消息处理）。
- 卸载组件时 `useWebSocketTopic` 在 `onUnmounted` 钩子里自动调用返回的
  取消订阅函数，仅从内存 `listeners` 集合中移除回调，不影响socket本身。

- **登录失效跳转**：

    已有约束，如无则记为前端缺口。

### 1. ollama 容器化方案 —— 为什么用 Modelfile + docker exec

**核心问题**：vllm 可以直接 `--model /models/<目录>` 指向一个 HF 格式权重
目录（safetensors + config.json），但 ollama **没有**"扫描目录直接服务
任意 .gguf 文件"这种模式。ollama 自己维护一套内容寻址的 blob 存储
（容器内 `/root/.ollama/models`），任何模型必须先通过
`ollama create <name> -f <Modelfile>` "注册"进这套存储——Modelfile 里
`FROM <path-to-gguf>` 这一行告诉 ollama 去读哪个 GGUF 文件、拷贝/链接进
blob 存储，之后才能 `ollama run/serve <name>` 或通过 HTTP
`/api/...`、OpenAI 兼容的 `/v1/chat/completions` 访问。

**因此采用的流程**（`core.docker_helper.OllamaManager`）：

1. **唯一共享容器**：与 vllm（每个模型一个独立容器/端口）不同，ollama
   引擎所有模型共享同一个 `gb10-ollama` 容器（`render_compose()` 渲染，
   官方 `ollama/ollama:latest` 镜像，GPU 透传配置与 vllm 容器一致——
   `deploy.resources.reservations.devices` 指定 nvidia runtime；额外挂载
   `/models:ro`（本地模型只读目录）与一个落在 `data_dir/ollama-data` 的
   持久化卷挂到 `/root/.ollama`，让通过 `ollama create` 注册过的模型在
   容器重启后依然存在）。
2. **ensure_running()**：`docker inspect` 判断容器是否已在运行，未运行则
   `docker compose up -d` 启动。
3. **render_modelfile(model_info, ollama_params)**：在模型目录下找
   `.gguf` 文件，构造 `FROM /models/<容器内相对路径>.gguf` 加上若干
   `PARAMETER num_ctx/num_predict/num_gpu/...` 行（值来自
   `param_advisor.recommend_ollama_params` 的 `default`）。
   **已知限制**：若一个模型被拆成多个分片 GGUF
   （如 `xxx-00001-of-00009.gguf`），目前只取第一个分片——ollama 的
   `FROM` 只接受单文件，分片合并需要额外的 `gguf-split --merge` 之类的
   预处理步骤，本阶段未实现，留作后续待办（已在代码注释中标注）。
4. **load_gguf_model()**：完整链路——
   `ensure_running()` → 把渲染好的 Modelfile 写到
   `data_dir/ollama-modelfiles/<name>.Modelfile` → `docker cp` 进容器
   `/tmp/<name>.Modelfile`（因为 `ollama create` 是在容器内部执行，
   宿主机文件本身对容器不可见）→ `docker exec gb10-ollama ollama create
   <name> -f /tmp/<name>.Modelfile`（用 `subprocess` 调用，超时设置
   30 分钟，大 GGUF 文件的 blob 拷贝可能较慢）→ 返回结构化结果
   （`ok/ensure_result/modelfile_path/modelfile_content/create_result`）。
5. **unload_gguf_model(name)**：`docker exec gb10-ollama ollama rm <name>`，
   不停止整个容器（其他模型可能仍在用）。

加载完成后，ollama 容器对外监听 `host_port`（默认 11434，可在加载请求里
通过 `host_port` 覆盖），OpenAI 兼容路径固定是
`/v1/chat/completions`——`debug_router`/`perf_router` 因此可以把 ollama
和 vllm 当作完全相同的下游来转发，不需要区分引擎。

## 1. CLI start/stop/restart/clean 真实实现

新增 `core/process_manager.py`，把"项目组件"按需求拆成三类来源：

- **(a) Web 后端**：`subprocess.Popen` 后台启动
  `uvicorn web.main:app --host {web_host} --port {web_port}`，Linux 下用
  `start_new_session=True` 实现 nohup 式脱离父进程（Windows 开发机上用
  `DETACHED_PROCESS | CREATE_NEW_PROCESS_GROUP` 近似），pid 写入
  `<data_dir>/web.pid`，日志写到 `<data_dir>/logs/web.log`。
  `web_backend_status()` 通过 pidfile + `os.kill(pid, 0)`（Windows 上用
  `tasklist`）判断是否在跑，而不是探测端口。`start_web_backend()` 启动后
  sleep 1.5 秒做一次健康自检——若进程已退出，直接把日志尾部内容打印给
  用户，而不是谎报"已启动"。
- **(b) SearXNG 容器**：复用既有 `core.docker_helper.SearxngManager` 与
  `DockerComposeManager`，固定指向 `deploy/searxng-compose.yml`。
- **(c) 已渲染的模型 compose 文件**：`model_compose_files()` 扫描
  `data/compose/*.compose.yml`，显式排除 `ollama.compose.yml`
  （`OLLAMA_COMPOSE_NAME` 常量）；`ollama_compose_file()` 单独处理这个共享
  stack（一个容器可能同时承载多个 `ollama create` 注册过的模型，不能因为
  某个 vllm 模型的 compose 文件也在同一目录就被一并 down 掉）。

四个命令的真实语义：

| 命令 | 行为 |
| --- | --- |
| `start` | env_doctor 检测（保留原交互确认逻辑）→ `start_web_backend()`（已运行则跳过）→ SearXNG `up -d`（若 compose 文件存在）→ 所有模型 compose 文件 `up -d`（ollama 共享 stack 优先）→ `_print_runtime_summary()` 打印端口/API/明文密钥 |
| `stop` | `stop_web_backend()`（SIGTERM + 等待最多 3 秒）→ 所有 compose 文件（含 ollama、searxng）执行 `docker compose stop`（**不** down，不删容器） |
| `restart` | 先完整执行一次 `stop_all()`，再执行一次 `start_all()`（两步独立打印，不是简单调用 stop 命令再调用 start 命令的 CLI 子进程拼接） |
| `clean` | 二次确认提示 → `stop_all()` → 对所有 compose 文件执行 `docker compose down`（删容器+网络）→ 删除 `data/web.pid`；**不删除** `model_scan_result.json`/`data/reports/`/`config/settings.yaml` |

`cli/main.py` 的四个子命令分别用 `try/except Exception` 包裹全部逻辑，
异常时打印 `[ERROR] ... 命令执行异常: <msg>` 并以 `sys.exit(1)` 退出，不会
裸抛 Python 堆栈跟踪给用户。`_print_component_results()` 统一渲染
`[OK]`/`[FAIL]` 前缀的执行结果列表。

**待真机验证**：Linux 下 `start_new_session=True` 的脱离效果、SIGTERM 后
uvicorn 是否能在 3 秒宽限内优雅退出（FastAPI 的 `on_shutdown` 钩子需要时间

`ollama create` 注册多个模型的共享容器是否符合预期（同一容器内的所有模型
会一起停止，这是已知且符合"stop=不删除"语义的设计）。

## 3. 新增"API 发布"看板

新增 `web/routers/api_directory_router.py`：`GET /api/api-directory`

## 4. 组件内存使用 + 运行观测真实数据

**内存采集**（`web/routers/components_router.py`）：新增
`_docker_stats_mem_by_container()`，单次调用
`docker stats --no-stream --format '{{.Name}}\t{{.MemUsage}}'`
拿到全部容器的内存占用，解析成 `{container_name: mem_mb}` 字典；
`_parse_mem_usage()` 处理 `"512.3MiB / 119.6GiB"` 这种格式，支持
B/KiB/MiB/GiB/TiB 单位换算。`list_components()` 与
`overview_router.get_overview()` 都改成"先批量查一次 docker
stats，再按 `container_name` 查字典"，不再对每个组件单独起一次
`docker stats` 进程——这是本次修复明确要求的性能点。任何解析失败的容器
其 `memory_usage_mb` 保持 `None`，不影响其余组件正常显示。

**运行观测**（新增 `web/routers/runtime_stats_router.py`，
`GET /api/models/{name}/runtime-stats`）：

- **启动参数**：`_read_compose_command()` 直接用 `yaml.safe_load` 读取
  `data/compose/{name}.compose.yml` 里 `services.<svc>.command` 列表原样
  返回，不重新拼装/推断参数。
- **内存占用**：复用上面同一个 `_docker_stats_mem_by_container()`。
- **KV Cache / 运行与等待请求数**：对 `loaded.engine == "vllm"` 且
  `host_port` 已知的模型，请求
  `http://127.0.0.1:{host_port}/metrics`（vllm 容器暴露的 Prometheus
  文本端点），`_parse_prometheus_text()` 用正则
  `^([a-zA-Z0-9_:]+)(\{[^}]*\})?\s+([0-9.eE+\-]+)\s*$`
  逐行解析出 `{metric_name: value}`，再用**子串匹配**（不是精确匹配）从
  中挑出包含 `num_requests_running`/`num_requests_waiting`/
  `gpu_cache_usage_perc` 的指标行。模型未加载、`/metrics` 连接失败或返回
  非 200 时，返回 `{"loaded": false, ...}` 或
  `metrics: null` + `metrics_message` 友好提示，不会让接口报 500。
- ollama 引擎当前没有等价的 `/metrics` 采集路径，`metrics_message`
  会直接说明"ollama 引擎暂未提供等价的 Prometheus /metrics 端点采集"。

**✅ 已验证（原阶段七遗留项）**：`METRIC_NAME_HINTS` 里的三个指标名
（`num_requests_running`/`num_requests_waiting`/`kv_cache_usage_perc`）
是 vllm 的 Prometheus 指标命名。已在真机确认：旧镜像
`nvcr.io/nvidia/vllm:26.05.post1-py3`(vLLM 0.21) 真实指标为
`vllm:kv_cache_usage_perc`（非最初猜的 `gpu_cache_usage_perc`）；升级到
`nvcr.io/nvidia/vllm:26.06-py3`(vLLM 0.22.1) 后这三个名字在容器内 vllm
源码中依然存在（已 grep 确认）。如后续版本子串匹配不上，仍可在真机执行
`curl http://127.0.0.1:8001/metrics` 复核，如果
版本不同导致子串匹配不上，需要回到 `runtime_stats_router.py` 的
`METRIC_NAME_HINTS` 字典调整匹配关键字（设计上已经做成字典化、子串匹配，
调整只需改这一处，不需要改动整体解析逻辑）。

`frontend/src/views/ObservabilityView.vue` 的"待后端提供 runtime-stats
接口"占位行已替换为真实表格行（模型占用内存/运行中请求数/等待中请求数/
KV Cache 使用率 + 折叠展开的启动参数原文），每 10 秒通过
`refreshAllRuntimeStats()` 刷新一次，与系统级 `metrics` topic 的刷新节奏
保持一致但走独立的 REST 轮询（因为这是按需对每个已加载模型单独请求，不
适合塞进广播式的 WS topic）。

## 5. SearXNG 网络代理配置持久化

`core/config.py` 的 `AppConfig` 新增 `searxng_proxy_url: str | None = None`
字段（YAML 落盘，`load_config`/`save_config` 不需要额外改动，已有的

`GET /api/settings` 读回之前保存的值，不再是纯前端临时状态。

跑一次真实代理转发请求确认参数名兼容（`requirements.txt` 锁定的版本号

# 阶段八～阶段十二：架构现状综述（截至 v1.5.0，2026-06-28）

> 本节汇总 v1.0.0 之后的重大架构变化，作为前述分阶段设计的现状修订。逐版本
> 细节见 Task_Tracking.md，逐项真机验证见 测试checklist.md「阶段九～阶段十二」。

## 1. 引擎统一：移除 ollama，全部走 vLLM（v1.1.0）
- 删除 `OllamaManager`、`recommend_ollama_params` 及各处 ollama 分支。
- safetensors 与单文件 GGUF 均由 `DockerComposeManager.render_compose()` 渲染为
  vLLM 容器：GGUF 的 `--model` 指向 `.gguf` 文件且强制 `--dtype float16`（vLLM 对
  GGUF 不支持 bfloat16）；多分片 GGUF / vLLM 不支持的量化类型（如 MXFP4）在
  `model_scanner` / `docker_helper` / `models_router` 三层拒绝。
- `--quantization` 不再传入，交给 vLLM 从 `config.json` 的 `quantization_config`
  自动检测（目录名里的 "NVFP4" 等只是展示标签，不是合法取值）。

## 2. 鉴权：全局 vllm_api_key（v1.1.0/v1.1.2）
- `AppConfig.vllm_api_key`（`sk-xxxx`，明文持久化），所有 vLLM 容器以 `--api-key`
  启动。`load_config()` 修复了「文件存在但缺新增字段时每次调用重新随机生成」的坑
  （与 secret_key 同类，否则已加载容器烧录的 key 与当前配置不一致→401）。
- vLLM 原生 `/v1/messages`(Claude) 也只认 `Authorization: Bearer`，**不认** x-api-key。

## 3. 对外 API 网关 `web/routers/gateway_router.py`（v1.4.0）
- 在 Web 端口(8000)统一暴露 OpenAI（`/v1/chat/completions`、`/v1/completions`、
  `/v1/embeddings`、`/v1/models`）与 Claude（`/v1/messages`、`/v1/messages/count_tokens`）。
- 按请求体 `model` 字段路由到对应已加载容器；找不到时回退到唯一的通用/嵌入模型。
- 用 `vllm_api_key` 鉴权，**同时接受 Bearer 和 x-api-key**（修正了 CherryStudio 等
  Claude 客户端用 x-api-key 连不上的问题），向上游一律转 Bearer；支持 stream 透传。
- 不挂 JWT 依赖（它是面向程序的数据面接口，非管理前端）。

## 4. 模型扫描：手动化 + 持久化缓存（v1.5.0）
- 扫描（stat 全树 + 解析 config.json/index.json + 读 GGUF 头）开销大，改为**显式**触发：
  Web `POST /api/models/rescan`（前端「重新扫描模型」按钮）或 `cli model_check`。
- `web/model_cache.py`：进程内缓存 + 磁盘 `data/model_scan_result.json` 为准，按文件
  mtime 自动重载（CLI 扫描后运行中的 Web 无需重启即可感知）；启动只 load 不 scan，
  仅首启无结果文件时 bootstrap 一次。
- 所有读路径（overview / models / api-directory / 网关 is_embedding 判断）改读缓存。

## 5. 模型文件完整性校验（v1.4.0/v1.4.1）
- `model_scanner._validate_model` 多维度静态校验（不涉及引擎加载）：格式/后缀、
  config.json 解析与关键字段、`*.index.json` 权重清单(weight_map)完整性、total_size
  大小/截断核对、空文件、GGUF 量化兼容性。`ModelInfo.valid` / `validation_errors`。
- 分片完整性以 index.json 的 weight_map 为权威（不靠重建文件名，规避 0/1 起始与补零
  位数差异导致的误报）；结果按目录签名缓存。
- 应用：前端高亮无效模型并禁止选择/加载、`models_router.load` 返回 422 门禁、CLI 标红。
- 可选 `verify_model_hashes()` + `cli model_check --verify-hash` 做 SHA256 深度校验
  （读完整文件，开销大，绝不进常规 scan）。

## 6. WebSocket 推送扩展与空闲优化（v1.3.0/v1.5.0）
- `MetricsCollector` 每 10s 除 `metrics` 外，额外广播 `overview` / `api_directory` /
  `runtime_stats` 三个整页快照，前端实时总览/运行观测/API发布仅首屏 REST 拉种子，
  其余靠推送，消除高频轮询。
- 优化：**无 WS 客户端连接时跳过全部页面快照广播**（其构建涉及 docker stats、逐模型
  httpx /metrics 抓取），空闲服务器仅保留廉价的 GPU/CPU 历史采样。

## 7. GPU / nvidia-smi 资源占用（审计结论，v1.5.0）
- 全代码不调用 `nvidia-smi` 子进程；GPU 指标仅 `background_tasks._read_gpu()` 一处用
  **pynvml(NVML 进程内绑定)**，每 10s 初始化一次后 3 个 C-API 调用，开销极低，已是
  比 nvidia-smi(每次 fork) 更优的方案，无需更换。

## 8. 其它页面/接口修正（v1.1.x~v1.3.0）
- 组件状态：固定列出 Web后端/前端/SearXNG/通用/嵌入 五行，含绑定 IP(0.0.0.0 局域网

- SearXNG 禁用 brave 引擎（公共端点限流刷错误日志）；并将 `server.base_url` 改为 `false`

## 10. 前端通信现状（v1.8.0）

- REST：用户操作类（登录/加载/卸载/重扫/调试/设置/性能测试运行删除/SearXNG 控制）+ 首屏种子
