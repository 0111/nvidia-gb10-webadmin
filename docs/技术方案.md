## 0. 项目背景与约束摘要

- **目标硬件**：NVIDIA DGX Spark（GB10 Grace Blackwell），ARM64 (aarch64)，128GB+ 统一内存。
- **推理引擎**：必须支持 NVIDIA 官方容器 `nvcr.io/nvidia/vllm:26.06-py3`（内含 vLLM 0.22.1 / CUDA 13.3，容器内 CUDA 版本高于宿主机，需 `cuda-compat-13-3` 兼容包 + `LD_LIBRARY_PATH` + 卷挂载配合），同时需支持 **ollama** 引擎。
- **模型目录**：`/home/spark/LocalModels/LLModels`，宿主机上**只读**，内含 safetensors / gguf 等多种格式、多种量化（NVFP4/FP8/GGUF Q8 等）的通用模型与 embedding/reranker 模型。
- **交付形态三件套**：自动部署脚本（环境 checklist + 部署）、CLI 工具（start/stop/restart/clean/model_check，含网络/内存/swap 等环境自愈逻辑）、Web 管理平台（9 个页面，重度依赖 WebSocket 实时推送）。
- **运行原则**：所有服务**不做持久化**，需人工手动启动；重要配置和状态需落地到目录文件，支持整机/项目快速迁移。
- **本地搜索**：需集成 SearXNG，并在 Web 端提供调试与代理配置能力。
- **团队工作模式（关键现实约束）**：开发与运行都在 DGX Spark 服务器（ARM64）上进行——现已改为通过 claude + SSH 直接登录服务器开发，代码的编辑、运行、调试、压测、git 提交均在服务器本机完成（不再有独立的 Mac/Windows 本地开发机做「只写不跑」的隔离）。这意味着：
  - 一切都直接跑在目标 ARM64 环境上，technology choice 的**可调试性、报错可读性、ARM64 原生支持成熟度**仍是首要权重（避免本地与目标环境差异带来的交付风险）。
  - 技术栈的 ARM64 原生支持成熟度、依赖体积、构建耗时，直接影响实际部署效率。

## 2. 技术选型方案 A / B / C

### 方案 A：Python（FastAPI） + Vue3 + docker compose + JSON/YAML 文件持久化

| 维度 | 选型 |
| --- | --- |
| 后端语言/框架 | Python 3.11 + FastAPI（同时提供 REST 与原生 WebSocket 支持） |
| 前端技术栈 | Vue3 + Vite + Pinia + 原生 WebSocket（或 socket.io-client） |
| 容器编排 | docker compose（CLI 工具 shell out 调用 `docker compose up/down`，模板用 Jinja2 渲染 compose 文件） |
| 配置/状态持久化 | 本地 JSON/YAML 文件（按组件分文件存放于项目数据目录），关键状态文件用原子写 + 文件锁 |
| 进程/容器生命周期管理 | FastAPI 后台任务 + subprocess 调用 docker compose / docker CLI；用 `docker stats`/`docker logs` 轮询 |

**优点**

- Python 生态对接 vllm/ollama 社区文档、HuggingFace config.json 解析（transformers 库）、NVML（pynvml）/GPU 监控库支持最成熟，模型智能识别（2.3.3 节的参数智能推断逻辑）开发效率最高。
- FastAPI 原生支持 `async def` WebSocket，与"指标推送、日志推送、模型加载日志流"等需求天然契合，几乎不需要额外网关。
- docker compose 是声明式的，配置即代码，与"不持久化、人工启动"的原则吻合——compose 文件本身就是可读的"配置快照"，便于排错和迁移。
- JSON/YAML 文件持久化无需额外数据库依赖，ARM64 上零安装成本，目录即整个状态，最符合"目录可整体迁移"的要求。
- 调试体感对"本地写代码、远程跑"的模式最友好：Python 报错栈清晰，FastAPI 有自动生成的 OpenAPI 文档，远程只需 `pip install` 或一个轻量虚拟环境，不存在交叉编译问题。

**缺点**

- Python 在高并发场景下单进程吞吐有限（GIL），但本项目的 Web 后端主要做编排和转发，真正的算力消耗在 vllm/ollama 容器内，FastAPI 自身的并发瓶颈影响有限。
- JSON/YAML 文件持久化在并发写入、复杂查询（如历史性能测试报告的多维筛选）场景下不如数据库；需要自行处理文件锁与版本兼容。
- docker compose 模板化渲染需要自行维护 Jinja2 模板，模板与 UI 表单字段需要保持同步，存在一定维护成本。

**适用场景**：团队以 Python 为主力语言、需要快速对接 AI/ML 生态库、希望以最低运维复杂度（无 DB、无 K8s）快速交付且强调"目录可迁移"的场景。**与本项目约束高度匹配**。

### 方案 B：Go（Gin/Echo + gorilla/websocket） + React + 自定义 Docker SDK 调用 + SQLite

| 维度 | 选型 |
| --- | --- |
| 后端语言/框架 | Go + Gin（REST）+ gorilla/websocket |
| 前端技术栈 | React + TypeScript + Zustand/Redux + 原生 WebSocket |
| 容器编排 | 不用 docker compose，直接用 Go 官方 `docker/docker` SDK（moby client）编程式创建/管理容器 |
| 配置/状态持久化 | SQLite（嵌入式，单文件数据库，配合 `modernc.org/sqlite` 纯 Go 驱动免 CGO 交叉编译问题） |
| 进程/容器生命周期管理 | Go 协程长期监听 Docker Events API，状态变更实时驱动 WebSocket 推送，无需轮询 |

**优点**

- Go 编译为单一静态二进制，部署到 ARM64 服务器只需复制一个文件，没有运行时依赖（不需要装 Python/Node 环境），符合"远程部署越简单越好"的诉求；CLI 工具与 Web 后端可以是同一个二进制的不同子命令。
- 使用 Docker SDK 直接编程式调用，能更精细地控制容器创建参数（环境变量、卷挂载、GPU runtime 配置），相比 compose 模板渲染更灵活，尤其适合"根据模型文件智能生成参数"这种动态性很强的场景。
- 监听 Docker Events 比轮询 `docker stats` 更省资源、延迟更低，对"实时总览""组件状态"页面的响应性更好。
- SQLite 单文件、零运维，比纯文件存储更适合"性能测试报告""日志元数据"等需要结构化查询的场景，文件本身依然可随项目目录整体迁移。
- Go 的并发模型（goroutine + channel）天然适合"多组件并行状态采集 + WebSocket 广播"的架构。

**缺点**

- **关键风险**：团队当前在本地（Mac/Win）只生成代码、不执行代码，所有验证都在远程 ARM64 服务器进行。Go 项目如果使用任何 CGO 依赖（包括部分 SQLite 驱动、部分系统监控库）将面临"本地无法验证编译结果是否能在 ARM64 上跑通"的问题；即使选用纯 Go 驱动规避 CGO，整个迭代调试节奏也会变成"改代码 -> 上传 -> 远程编译 -> 远程运行 -> 看日志 -> 再改"，单次反馈循环比 Python/Node 脚本语言更长。
- Go 在 AI/ML 生态（解析 HuggingFace config.json、读取 safetensors header、识别量化格式等）的库支持远不如 Python 成熟，"模型智能识别与参数推荐"这一核心功能需要大量手写解析逻辑，开发成本显著高于方案 A。
- React + TypeScript 的工程化程度高，但相应的构建链（Node/npm/Vite/webpack）也需要在远程或 CI 上构建产物，多了一层"前端构建是否在 ARM64 上顺利完成"的不确定性（虽然前端构建产物本身是平台无关的静态资源，可在任意机器构建后再上传，风险低于后端二进制交叉编译）。
- Go 团队学习曲线/人力如果本来就以 Python/JS 为主，引入第三种语言会增加维护成本。

**适用场景**：追求最小运行时依赖、极致部署简洁性、且团队具备 Go 工程能力、可以接受较长的远程调试反馈循环的场景。在"本地不执行代码"的约束下，该方案的调试效率劣势会被放大。

### 方案 C：Node.js（NestJS） + React + Docker Compose v2 + SQLite（落盘）+ 内存缓存

| 维度 | 选型 |
| --- | --- |
| 后端语言/框架 | Node.js + NestJS（内置 WebSocket Gateway 装饰器） |
| 前端技术栈 | React + TypeScript（与后端同语言生态，类型可共享） |
| 容器编排 | docker compose v2（NestJS 通过 child_process 调用 `docker compose`，复杂场景辅以 dockerode 库做细粒度操作） |
| 配置/状态持久化 | 内存缓存（指标类高频数据）+ SQLite/JSON 落盘（配置类低频数据），定期 flush |
| 进程/容器生命周期管理 | NestJS 后台任务（`@Cron`/interval）轮询容器状态 + dockerode 监听事件，统一通过 Gateway 广播 |

**优点**

- TypeScript 全栈统一（前后端同语言），类型定义（DTO/接口）可在前后端共享，减少前后端契约不一致的 bug，对"参数配置项极多"（vllm 30+ 参数）的表单类页面尤其有利，开发效率较高。
- NestJS 的 WebSocket Gateway、依赖注入、模块化架构对"多页面、多模块（模型配置/调试/日志/性能测试）"这种中大型单体应用的可维护性较好，工程规范性强于裸 Express/FastAPI。
- "内存缓存 + 定期落盘"的混合持久化方式可以减少高频指标（10 秒级刷新的 CPU/GPU/内存曲线）对磁盘的写压力，性能较纯文件持久化更优。
- Node.js 在 ARM64 上的官方发行版成熟稳定，`npm install` 跨平台依赖问题比 Go 交叉编译少，比 Python 个别带 C 扩展的库（如某些 GPU 监控库）兼容性更好。

**缺点**

- Node.js/NestJS 生态对接"解析模型文件、识别量化格式、推断 vllm/ollama 参数"这类 AI 基础设施相关任务缺乏现成库，几乎所有模型智能识别逻辑都要手写或者反向调用 Python 脚本（这会让架构退化为"Node 套 Python 子进程"的双语言架构，复杂度不降反升）。
- NestJS 学习曲线相对 FastAPI/Express 更陡（装饰器、模块、依赖注入概念），团队如果不熟悉 Angular 风格的工程范式，初期开发速度会受影响。
- "内存缓存 + 定期落盘"引入了缓存与落盘数据一致性问题：服务异常退出（例如人工 kill 后未走正常 shutdown hook）可能丢失最近一段时间未落盘的状态，与"配置数据需要可靠落盘以支持快速迁移"的要求存在一定张力，需要额外实现优雅关闭逻辑兜底。
- Node.js 单线程事件循环在做大量 CPU 密集型操作（例如解析数百个 safetensors 文件头、批量计算模型大小）时不如多进程/多协程模型友好，需要额外引入 Worker Threads，增加架构复杂度。

**适用场景**：团队前后端工程师以 TypeScript/JavaScript 为主，重视前后端类型一致性和大型单体应用的工程规范性，且能接受为 AI 相关能力额外引入 Python 辅助脚本的场景。

## 4. 产品经理推荐方案（建议，非最终决定）

**初步建议：方案 A（Python/FastAPI + Vue3 + docker compose + JSON/YAML 文件持久化）。**

理由：

1. 本项目的核心难点之一是"模型智能识别与参数推荐"（2.3.3 节中详细列出的 vllm/ollama 30+ 参数的智能匹配逻辑），这部分工作量大、逻辑复杂，Python 在 HuggingFace 生态、safetensors/GGUF 文件头解析、NVML GPU 监控等方向的库支持远超 Go/Node，可以显著降低研发经理的实现成本和踩坑概率。
2. 团队的现实工作模式是"本地只写代码、远程才能跑"，这意味着每一次调试都有较高的远程往返成本。Python 作为解释型语言，远程部署只需保证依赖装好即可运行，报错信息和调试体验对这种工作模式最友好，能最大程度缩短"写代码 -> 远程验证 -> 改代码"的反馈环。Go 的交叉编译/二进制分发优势在"本地不能执行代码"的约束下反而成了劣势，因为本地完全不能预先验证编译是否成功、是否存在 ARM64 特定问题。
3. "不持久化、目录可整体迁移"的要求与纯文件配置存储（JSON/YAML）天然契合：整个项目状态就是文件系统的一个目录树，无需额外的数据库版本管理、备份、迁移脚本，最符合"快速迁移"的诉求。
4. docker compose 的声明式特性使得"部署脚本""环境 checklist"这两个需求可以共享同一份 compose 模板和环境检测脚本，CLI 与 Web 后端复用同一套 Python 核心库（建议拆分为独立的 `core` 包），减少逻辑分叉风险。

需要注意的潜在短板（建议在确认方案前一并评估）：

- FastAPI 单进程的吞吐量在"多用户同时高频访问"场景下不是最优解，但本项目是面向运维管理员的内部工具，并发用户数预期很低，该短板影响有限。
- 文件持久化在性能测试报告积累较多后，查询效率会下降，可以在后续迭代中按需引入 SQLite 作为"报告类数据"的补充存储（不影响配置类数据仍用文件），属于渐进式优化，不影响现在的方案选择。

**再次强调：以上是产品经理基于现有需求和约束的建议，最终技术选型需要项目经理组织研发经理、测试经理共同评审（参见 Project_Task.md 2.2 节工作流程），并由人工最终确认。**
