> 对应版本 v2.1.0。完整版本历史见 [Task_Tracking.md](Task_Tracking.md)。

## 1. 环境与目录

- 服务器：DgxSpark (192.168.199.4)，用户 spark
- 项目目录：`/home/spark/nvidia-gb10-manager`
- 模型只读目录：`/home/spark/LocalModels/LLModels`
- 开发方式：通过 claude + SSH 直接登录该服务器开发，编辑/构建/运行/测试/git 提交均在服务器本机完成（不再有独立 Windows 开发机）

## 2. 首次部署

```bash
ssh spark@192.168.199.4
cd /home/spark/nvidia-gb10-manager
bash deploy/bootstrap.sh        # 环境checklist + 创建venv + 装依赖 + 生成 config/settings.yaml
grep -E 'admin_|vllm_api_key|secret_key' config/settings.yaml   # 记录管理员账号密码与API密钥（均明文）
```

## 3. 启动 / 停止服务（手动管理，不做持久化自启）

统一通过 CLI 一条命令管理全部组件：

```bash
cd /home/spark/nvidia-gb10-manager
source .venv/bin/activate          # 重要：用项目 venv，否则 uvicorn 等依赖找不到

python -m cli.main start           # 一次性启动全部组件，打印端口/URL/账号密码
python -m cli.main stop            # 停止全部组件（不删容器，下次 start 可快速恢复）
python -m cli.main restart         # 等价于 stop 再 start
python -m cli.main clean           # 停止并删除容器/网络/pid（保留扫描结果/性能报告/配置）
python -m cli.main model_check                 # 扫描并校验模型，写入 data/model_scan_result.json
python -m cli.main model_check --verify-hash   # 额外做 SHA256 深度校验（仅对带哈希清单的模型，较慢）
```

`start` 启动顺序（顺序有意为之：模型容器先于 Web 后端，确保后端启动时对账能看到容器）：
1. 环境 checklist（cuda-compat / 网卡协商 / drop_caches / swap）
2. SearXNG 容器（`deploy/searxng-compose.yml`）
3. `data/compose/` 下所有已渲染的模型 compose 文件
4. Web 后端（`uvicorn web.main:app`，pidfile 管理；非 venv 调用也能起，会优先用 `.venv/bin/python`）
5. 前端：`frontend/dist` 不存在时自动 `npm install && npm run build`，再 `vite preview` 对外服务

> 注：ollama 已彻底移除；不再有 ollama 共享容器这一步。

## 4. 登录 Web 管理界面

浏览器访问 `http://<服务器IP>:4173`，用 `config/settings.yaml` 的 `admin_username` /
`admin_password` 登录（项目内所有密钥均明文显示，不做遮掩）。

## 5. 扫描并校验模型

- 模型扫描会读取磁盘文件、开销较大，因此为**手动**操作：
  - Web：「通用大模型配置 / 嵌入式模型配置」页点击「重新扫描模型」按钮，页面显示「上次扫描时间」。
  - CLI：`python -m cli.main model_check`。
- 扫描会做多维度文件校验：config.json 解析、`*.index.json` 权重清单完整性（缺分片）、
  total_size 大小/截断核对、空文件、GGUF 量化兼容性。**校验未通过的模型会高亮、禁止加载**。
- 结果持久化在 `data/model_scan_result.json`，被总览/模型列表/API发布等所有页面复用。

## 6. 加载一个模型（核心操作）

1. 在「通用大模型配置」或「嵌入式模型配置」页选择模型 → 系统自动读取推荐 vLLM 参数。
2. 确认/修改参数（如 max_model_len 下拉、gpu_memory_utilization；勾选 enable_auto_tool_choice
   后可选 tool_call_parser）。
3. 点击加载 → 后端渲染 `data/compose/<模型名>.compose.yml` 并 `docker compose up -d`。
4. 通用模型固定容器 `gb10-vllm-general`(端口 8001)，嵌入模型 `gb10-vllm-embedding`(端口 8002)；
   **每类只能同时加载 1 个**，已加载时再加载同类会被拒绝（提示先卸载）。
5. 加载完成（大模型权重加载约 1～3 分钟）后即可调用；不用时务必卸载（总览页点卸载）释放显存。

## 7. 对外 API 调用（OpenAI / Claude）

统一入口（推荐）：`http://<服务器IP>:8000/v1`，鉴权用 `vllm_api_key`（Bearer 或 x-api-key 均可）。

```bash
# OpenAI 兼容
curl http://192.168.199.4:8000/v1/chat/completions \
  -H "Authorization: Bearer <vllm_api_key>" -H "Content-Type: application/json" \
  -d '{"model":"<已加载的通用模型名>","messages":[{"role":"user","content":"你好"}]}'

# Embeddings（需已加载嵌入模型）
curl http://192.168.199.4:8000/v1/embeddings \
  -H "Authorization: Bearer <vllm_api_key>" -H "Content-Type: application/json" \
  -d '{"model":"<已加载的嵌入模型名>","input":"hello"}'

# Claude 兼容（Anthropic Messages）
curl http://192.168.199.4:8000/v1/messages \
  -H "x-api-key: <vllm_api_key>" -H "Content-Type: application/json" \
  -d '{"model":"<已加载的通用模型名>","max_tokens":256,"messages":[{"role":"user","content":"你好"}]}'
```

- CherryStudio 等客户端：用「OpenAI 兼容」类型填 BaseURL=`http://192.168.199.4:8000` + Bearer 密钥
  最稳；用 Claude 类型(x-api-key)连网关也可（网关两种鉴权头都接受）。
- 直连容器端口 8001/8002 仅用于排障；注意原生 vLLM 只认 Bearer，不认 x-api-key。
- 「实时总览 → API 发布」页列出全部对外接口与密钥，并有「检测API健康状态」按钮做主动探测。

**Claude Code / claude CLI 连本地模型**（macOS 示例）：

```bash
export ANTHROPIC_BASE_URL="http://192.168.199.4:8000"
export ANTHROPIC_AUTH_TOKEN="<vllm_api_key>"      # 即 settings.yaml 里的 vllm_api_key
export ANTHROPIC_MODEL="<已加载的通用模型名>"        # 如 Qwen3.6-35B-A3B-NVFP4
claude
```

网关会把 Claude 的 `/v1/messages` 请求体清洗后转发给 vLLM（自动把混入 messages 的 system
角色并入顶层 system 字段，规避 vLLM 的 400 校验错误），流式/非流式均支持。务必先在 Web 端
加载好 `ANTHROPIC_MODEL` 指定的通用模型（未加载会回退到当前唯一已加载的通用模型）。

## 8. SearXNG 本地搜索

- 容器 `gb10-searxng`，端口 8080，`/search?q=...&format=json` 返回 JSON。
- 在「实时总览 → 组件状态」可对 SearXNG 启动/停止/重启。
- 已禁用 brave 引擎（公共端点频繁限流刷错误日志）；改 `deploy/searxng-settings.yml` 后需
  `docker restart gb10-searxng` 才生效。

## 9. 已知限制

- 多分片 GGUF 需先用 `tools/gguf_merge_shards.py` 合并为单文件再加载（vLLM 只支持单文件 GGUF）。
- MXFP4 量化的 GGUF 不被 vLLM 0.22 支持（如 GPTOSS-120B-...-MXFP4），扫描会判为无效并拒绝加载。
- 性能测试为自定义轻量级并发吞吐测试，未对接 MLPerf/LLMPerf 等标准 benchmark。
- 运行观测的「模型级 tok/s」已实现：从 vLLM /metrics 的 prompt/generation token 计数器差分得
  prefill/decode tok/s，与系统级指标同样持久化为近 1 天趋势图，并显示峰值（最大值）。
