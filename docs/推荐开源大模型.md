# 本地开源大模型 · 按场景推荐清单

> 平台：NVIDIA DGX Spark（GB10 Grace Blackwell，**128GB 统一内存**，带宽 ~273GB/s）。
> 生成时间：2026-07-02，依据本机 `/home/spark/LocalModels/LLModels` 实际扫描结果（29 个通用 + 5 个嵌入/重排）。
> 本清单只推荐**本机已有**的模型；每条给出理由、显存占用与加载要点。

## 平台选型原则（先读）

- **优先量化版**：GB10 是统一内存 + 带宽瓶颈型，量化能同时省显存、提 decode 速度。
  - **NVFP4（~4bit）**：最省显存、decode 最快；但数值最脆，个别「aggressive 合并量化」偶发输出乱码/卡死（满屏 `!`），必要时勾选 `enforce_eager`。
  - **FP8**：比 NVFP4 大一档，**数值最稳**，速度/质量均衡——追求稳定选它。
  - **未量化（bf16）**：最准但最占显存、decode 最慢；仅在显存充裕且极重精度时用。
- **MoE（如 A3B = 激活 3B）**：全量权重常驻内存，但每 token 只算激活的少量专家 → **decode 快**，非常适合 GB10。
- **共存预算**：通常「1 个通用/多模态 + 1 个嵌入」同时加载。两者 `gpu-memory-utilization` 之和建议 ≤ 0.7，**给系统与 `/dev/shm` 留 ~30GB 余量**（余量不足会 OOM/不稳）。
- **避免**：多分片未合并 GGUF、MXFP4 GGUF（vLLM 加载器不支持）——见文末「不建议/需注意」。

## 九、最不推荐清单（避坑，按严重度排序）

### 🔴 第一档：基本别用
| 模型 | 问题 |
|---|---|
| **GPTOSS-120B-Uncensored-HauhauCS-Aggressive**（GGUF MXFP4，61GB） | ❌ **最不推荐**。扫描 `valid=False`，vLLM 的 GGUF 加载器不支持 MXFP4 量化，**根本加载不了**。 |
| **`*-Uncensored-HauhauCS-Aggressive-NVFP4`**（如 Qwen3.6-35B-A3B-Uncensored-…-NVFP4） | aggressive 合并量化**数值脆**，偶发满屏 `!`/引擎卡死；用则需「卸载重载 + 勾 `enforce_eager`」，或改用对应 **FP8** 版。 |
| **GLM-4.7-Flash-Uncensored-HauhauCS-Aggressive**（GGUF FP16，56GB） | Dense GGUF、体积大、`tools=False`、aggressive 无审查合并，优化度与稳定性都差。 |

### 🟠 第二档：纯浪费显存（有更优量化同款，别选未量化版）
在 GB10 上又大又慢，且本机都有 NVFP4/FP8 同款：

| 不推荐（未量化 bf16） | 占用 | 应改用 |
|---|---|---|
| Qwen3.6-35B-A3B | 67GB | Qwen3.6-35B-A3B-**FP8**(35G) / **NVFP4**(22G) |
| DeepSeek-R1-Distill-Qwen-32B | 61GB | DeepSeek-R1-Distill-Qwen-32B-**NVFP4**(19G) |
| Qwen3-VL-30B-A3B-Instruct / Thinking | 各 58GB | Qwen3-VL-30B-A3B-Thinking-**NVFP4**(18G) |
| GLM-4.7-Flash | 58GB | GadflyII-GLM-4.7-Flash-**NVFP4**(19G) / GLM-4.7-Flash-**FP8**(30G) |

（这些未量化版还常见 `tools=False`，函数调用也不便。）

### 🟡 第三档：老 / 受限，有更好选择
| 模型 | 问题 | 更优替代 |
|---|---|---|
| **QwQ-32B**（61GB，未量化） | 上下文仅 **40960**、`tools=False`、体积大 decode 慢 | DeepSeek-R1-Distill-Qwen-32B-NVFP4 |
| **Qwen3.6-27B**（52GB，Dense 未量化） | 27B 稠密又大又慢，能力/效率不如 MoE 量化版 | Qwen3.6-35B-A3B-NVFP4（激活 3B，22GB） |
| **老 GGUF 单文件**（Gemma-4-31B-JANG-Q8、Gemma-4-E4B-Q8、Qwen3-14B-BaronLLM-Q8） | 相较 safetensors NVFP4/FP8 优化/生态弱；多分片需先合并 | 对应 safetensors 量化版（BaronLLM 若专用安全微调可保留） |

### 一句话
最该避开：**GPTOSS-120B MXFP4（加载不了）> `*-Aggressive-*` 无审查合并（不稳）> 一切"未量化大模型"（有量化同款时纯浪费）> QwQ-32B / Qwen3.6-27B（老/受限）**。

## 通用注意事项

- **`--dtype`**：NVFP4/GB10 上用 `auto`（bf16），**不要强设 float16**（易溢出→NaN/`!`）。
- **多分片 GGUF**：需先用 `tools/gguf_merge_shards.py` 合并为单文件才能加载。

> 显存换算参考：128GB 统一内存约 121.7GiB 可用；同时加载两模型时，二者权重 + KV + 激活 + 系统 + `/dev/shm` 共抢，务必留 ~30GiB 余量。
