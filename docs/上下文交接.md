> **v1.0.0 之后的重大变化（务必先读这一段）**：
> 1. **ollama 引擎已彻底移除**（v1.1.0）。所有本地模型——safetensors 和 GGUF
>    一律走 vllm 容器加载。原"多分片GGUF需ollama"的说法作废；多分片GGUF用
>    `tools/gguf_merge_shards.py` 合并为单文件后由 vllm 加载。GGUF 必须 `--dtype
>    float16`（bfloat16 会崩），且 vllm 0.22 的 GGUF 加载器不支持 MXFP4 量化。
> 2. **新增全局 `vllm_api_key`（sk-xxxx）**：所有 vllm 容器用 `--api-key` 启动，
>    客户端调 /v1/* 需带 `Authorization: Bearer <key>`（或 Claude 风格 `x-api-key`）。
>    该 key 在 settings.yaml 持久化（v1.1.2 修了"每次 load_config 重新随机生成"的bug，
>    与 secret_key 同类坑）。**注意：模型的 compose 文件在加载时把当时的 key 烧进去了，
>    如果之后 key 变了，旧容器会用旧 key → 客户端 401。重新加载该模型即可同步最新 key。**
> 3. **vllm `--quantization` 改为不传**（v1.1.2），交由 vllm 从 config.json 的
>    quantization_config 自动检测。绝不要把目录名里的"NVFP4"等展示标签直接当
>    `--quantization` 值传（不是合法取值，会启动失败）。
> 4. **单模型加载限制**（v1.2.0）：通用/嵌入各只能同时加载1个（固定容器名
>    gb10-vllm-general / gb10-vllm-embedding）。该类已加载时再加载别的会被后端
>    409 拒绝，提示先卸载；重新加载同名模型是允许的（幂等重启，也用于同步 key）。
> 5. **API 健康主动检测**（v1.2.0）：`GET /api/api-directory/health-check` 真实探测
>    各端点，实时总览→API发布页有"检测API健康状态"按钮。
> 6. **版本发布脚本**（v1.2.0）：`bash tools/release.sh vX.Y.Z "说明"`，标准化
>    提交+打tag+追加 Task_Tracking.md 版本行。
> 7. **CLI 健壮性**（v1.2.0）：`python3 -m cli.main start` 即使不在 venv 里也能正常
>    起 web 后端了（process_manager 优先用 `.venv/bin/python` 跑 uvicorn）。
> 8. **vllm 鉴权统一走 Bearer**（v1.3.0 确认）：vllm 0.22 的 `--api-key` 中间件
>    对所有路由（含 Anthropic 兼容的 `/v1/messages`）只认 `Authorization: Bearer <key>`，
>    **不认** Anthropic 标准的 `x-api-key`（实测发 x-api-key 会 401）。所以

## 1. 项目是什么

NVIDIA DGX Spark / GB10 Grace Blackwell 平台的本地大模型管理工具。核心功能：扫描并校验本地模型文件、智能推荐vllm启动参数、通过Web界面和CLI管理模型加载/卸载、OpenAI+Claude兼容对外API网关、SearXNG本地搜索集成、系统资源监控与运行观测。（注：ollama 已移除，统一 vLLM。）

设计与需求的权威来源是 [技术方案.md](技术方案.md)（已确认的技术选型）与 [研发方案.md](研发方案.md)（分阶段的详细设计与真机验证）。（注：早期有一份原始需求文档 `Project_Task.md`，未纳入本仓库；代码注释里偶有「per Project_Task.md X.Y」的设计出处引用，仅作历史溯源。）

## 3. 技术方案（已确认，不要重新讨论）

详见 [技术方案.md](技术方案.md)（已确认版本，"v1.1 项目经理 已确认：采用方案A"）。

**方案A**：Python 3.11 + FastAPI（Web后端） + Vue3 + Vite + Pinia（前端） + docker compose（容器编排） + JSON/YAML文件持久化（无数据库）。CLI与Web后端共享同一套`core/`核心库。

选择理由（已确认，不要重新论证）：模型解析/参数推荐是核心难点，Python生态最成熟；团队"本地写码、远程才能跑"的现实约束下，Python调试反馈链路最短；纯文件持久化最符合"目录可整体迁移"的要求。

## 6. 已知限制（不是bug，已确认不处理）

| 限制 | 说明 |
|---|---|
| 多分片GGUF需先合并 | vllm 仅支持单文件 GGUF，多分片需用 `tools/gguf_merge_shards.py` 合并（已不再依赖 ollama）。后端三层校验会拒绝直接加载多分片 GGUF 并给出提示。 |
| MXFP4 量化的 GGUF 无法加载 | vllm 0.22 的 GGUF 加载器不支持 MXFP4（如 `GPTOSS-120B-...-MXFP4.gguf`），model_scanner/docker_helper/models_router 三层拒绝。 |
| 性能测试未对接MLPerf/LLMPerf国际标准benchmark | 只做了自定义轻量级并发吞吐测试 |
| 高并发/多用户登录场景未专项测试 | 内部工具，用户量低，非重点 |

## 8. 重要的调试经验/坑（避免重复踩）

1. **Git Bash路径转换坑（已不再相关）**：早期 Windows 开发机用 Git Bash 调 `tools/remote.py` 时，远程路径会被 MSYS 转换，需加 `MSYS_NO_PATHCONV=1`。现已直接在服务器 Linux shell 上开发，不再有此问题；仅作历史记录保留。
2. **`Path.stem`不能正确处理`.compose.yml`这种双后缀文件名**：只会去掉最后一个`.yml`，留下`.compose`尾巴。要用`path.name[:-len(".compose.yml")]`。
3. **vllm 0.22 版本参数（0.21+ 起）**：`--task embed`已废弃报错"unrecognized arguments"，必须用`--convert embed`+`--runner pooling`。每次vllm镜像升级都要重新确认`vllm serve --help=<关键字>`里的真实参数名，不要凭经验/旧文档假设。本次升级 26.05.post1(0.21)→26.06(0.22.1) 已用真机核对参数：tool-call-parser 多了 apertus、quantization 多了 auto_gptq、kv-cache-dtype 多了 nvfp4、tokenizer-mode 多了 hf。
4. **vllm容器的command渲染规则**：容器entrypoint是裸`exec "$@"`无默认CMD，compose的command列表必须以`["vllm","serve","--model",路径,...]`开头，且每个flag和它的value必须是**独立的列表项**（不能写成`"--flag value"`一个字符串），否则容器启动即崩。
5. **26.06 镜像的 /v1/* 接口必 500（已修，v2.2.1）**：基础镜像 `nvcr.io/nvidia/vllm:26.06-py3` 内 `fastapi==0.137.1` 新增了无 `.path` 属性的内部路由类 `_IncludedRouter`，而 `prometheus_fastapi_instrumentator==8.0.0` 的中间件会对每个非排除请求读 `route.path` → 抛 `AttributeError: '_IncludedRouter' object has no attribute 'path'`。**现象极具迷惑性**：模型加载正常、`/metrics` 200（在 excluded_handlers 里），但 `/v1/chat/completions`、`/v1/embeddings` 等一律 500——很容易误判成"参数不对/gpu_memory 设置问题"，其实与启动参数无关。修法：`deploy/vllm-patch.Dockerfile` 用 sed 把 instrumentator 的 `route.path` 改成 `getattr(route,"path",None)`，构建出 `gb10-vllm:26.06-py3-patched`，`config.vllm_image` 默认指向它（bootstrap 第[5/5]步幂等构建）。已真机验证：补丁镜像上 chat/embeddings/网关全部 200。
5. **docker compose的`gb10-vllm-general`/`gb10-vllm-embedding`是固定共享容器名**，只能同时跑一个通用模型+一个嵌入模型。加载新模型前要清理同容器名的旧compose文件，否则磁盘上堆积的过期文件会让"reconcile"逻辑误判。
6. **判断"当前实际加载的是哪个模型"不能靠文件名猜，要用`docker inspect -f '{{.Args}}'`读容器真实的`--served-model-name`**——这是web后端`registry`纯内存、重启后必须靠`web/state.py`的`reconcile_from_disk()`重建状态时踩过的坑。
7. **`docker stats`输出的内存单位（KiB/MiB/GiB/TiB）都以"B"结尾**，按字典遍历做后缀匹配时如果"B"排在前面会被错误地优先匹配截断，必须按后缀长度从长到短检查。
8. **`npx`是包装进程会再fork真正的子进程**，SIGTERM只杀包装层、子进程（如`vite preview`）会变成孤儿继续跑，必须对整个进程组`os.killpg()`发信号才能真正杀干净。
9. **`core.config.load_config()`如果配置文件不存在，必须立即落盘**，否则每次调用都会生成不同的随机密钥/密码，导致JWT签发和校验不一致、登录必然失败（这是v0.2.0的严重bug）。
10. **`vite preview`是纯静态文件服务器，没有`/api`代理**——前端(4173)和后端(8000)实际是跨端口部署，但`frontend/src/api/client.ts`和`stores/ws.ts`一开始默认假设"同源"（用`window.location.host`拼URL），导致登录请求打到前端自己的端口404，表现得像"账号密码不正确"（v1.0.1修复，改为默认拼`${hostname}:8000`）。**如果用户反馈"登录/WebSocket不通但密码确认是对的"，先查这里**，本机`curl 127.0.0.1`测后端API永远是对的，因为那是直接打后端，不会暴露这个前端跨端口的问题——必须模拟"从前端域名+端口发起跨域请求"才能复现。

## 9. 工作流程约定（CLAUDE.md + 本项目实践）

- 项目走"产品经理→项目经理确认→研发经理写代码→测试经理验证"的角色化流程（详见Project_Task.md 2.2节），但实践中是同一个AI会话内按角色切换完成。
- **每次往服务器推送代码都要在服务器git仓库里commit，commit信息要写清版本号(语义化递增)+本次改了什么+是否真机验证过**。
- **重要原则：代码改完不算完成，必须在服务器上真实跑起来、用真实数据验证过才算完成**。本项目历史上多次出现"代码逻辑看似正确，但真机一跑就报错"的情况（见上面"坑"列表），所以任何新功能/修复都要在ARM64+GPU服务器上实测，不能只看本机静态检查（语法检查/npm build只是必要不充分条件）。
- 文档维护：[Task_Tracking.md](Task_Tracking.md)记录版本历史，[测试checklist.md](测试checklist.md)记录测试用例和发现的bug，[操作手册.md](操作手册.md)给最终用户看的操作说明，[研发方案.md](研发方案.md)记录架构设计细节（按"阶段一"到"阶段七"分章节追加，没有重写）。**新会话如果要继续开发，应该先读这4个文档而不是猜测现状**。

## 10. 如果用户接下来说"继续完成XX工作"，应该怎么做

1. 先读本文档 + [Task_Tracking.md](Task_Tracking.md)的版本表，了解现状。
